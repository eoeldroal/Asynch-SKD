## Implemented Pieces

### Protocol client

- File: [`web_osgym_protocol.py`](/home/sogang_nlpy/verl/verl/experimental/agent_loop/web_osgym_protocol.py)
- Sends `POST /` requests with `op=start|action|reward`.
- Uses wire field `session_id`.
- Parses optional base64 PNG image into a PIL image.

### Persistent environment tool

- File: [`web_osgym_tool.py`](/home/sogang_nlpy/verl/verl/tools/web_osgym_tool.py)
- Exposes the model-facing `computer` tool with `actions: [...]`.
- Keeps environment state in `_instance_dict`.
- Preserves `task_id`, `session_id`, `include_a11y`, and cached reward for each tool instance.
- Treats `DONE` / `FAIL` as terminal actions and requires them to be standalone.

### Loop helpers

- File: [`web_osgym_loop_mixin.py`](/home/sogang_nlpy/verl/verl/experimental/agent_loop/web_osgym_loop_mixin.py)
- Starts the remote environment session.
- Allocates runtime-owned `web_osgym_session_id`.
- Restores local tool instance state when partial async SKD state resumes.
- Fetches final reward once at termination.

### SKD loop

- File: [`web_skd_agent_loop.py`](/home/sogang_nlpy/verl/verl/experimental/agent_loop/web_skd_agent_loop.py)
- Registered as `web_skd_agent`.
- Requests `include_a11y=True`.
- Maintains separate student and teacher prompt streams.
- Normal visual observation:
  - student sees image
  - teacher sees image + text/a11y
- Image-less failure observation:
  - student and teacher both see text
- Final environment reward is copied into `AgentLoopOutput.reward_score`.

### Mock server/client

- Server: [`web_osgym_mock_server.py`](/home/sogang_nlpy/verl/async_skd/mock_server/web_osgym_mock_server.py)
- Client: [`web_osgym_mock_client.py`](/home/sogang_nlpy/verl/async_skd/mock_server/web_osgym_mock_client.py)
- Purpose: verify the protocol before attaching real WebGym / Omnibox.
- The server validates session lifecycle and writes JSONL request logs.

## Critical Invariants

1. `task_id` identifies the environment task.
2. `session_id` identifies one environment trajectory.
3. `session_id` is generated by the agent-loop/runtime side, not by the dataset or server.
4. A trajectory must reuse the same `session_id` for `start -> action* -> reward`.
5. The model emits a `computer` tool call with `actions: [...]`; it does not emit protocol JSON directly.
6. `DONE` and `FAIL` are model-facing actions but terminal from the loop perspective.
7. Async SKD scheduler/trainer layers should not special-case Web / OS Gym.

## Current Verification Flow

### 1. Protocol smoke

Start the mock server:

```bash
cd /home/sogang_nlpy/verl
conda activate skd

python async_skd/mock_server/web_osgym_mock_server.py \
  --host 127.0.0.1 \
  --port 18000 \
  --log-path logs/mock_web_osgym_requests.jsonl
```

Run the mock client:

```bash
python async_skd/mock_server/web_osgym_mock_client.py \
  --base-url http://127.0.0.1:18000 \
  --session-id 777 \
  --task-id 12345
```

Expected log sequence:

```text
start -> action(CLICK) -> action(DONE) -> reward
```

All entries must use the same `session_id` and `task_id`.

### 2. Tool / loop integration

After protocol smoke passes, point `WebOsGymTool` at the mock server and check:

- `start` is called before generation
- actions are logged as sent
- terminal action triggers reward fetch
- reward reaches `AgentLoopOutput.reward_score`

### 3. Real WebGym / OSWorld

Only after mock protocol and loop integration pass should the real environment stack be attached. At that point, remaining failures are more likely to be environment readiness, browser session, Omnibox, or task-specific issues.
